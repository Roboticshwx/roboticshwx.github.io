LLM + Robotics

基础知识

The way we program robots today is rigid! They can’t be flexibly re-programmed by everyday users.

传统机器人执行pick and place任务的流程:

  1. 利用cv算法检测出要抓取的目标物体;
  2. 计算机械臂该用什么姿势抓取物体;
  3. 移动机械臂到抓取姿势并抓取
  4. 根据要求移动机械臂到指定位置;
  5. 计算机械臂放下物体的姿势并放下;

传统机器人控制中存在的弊端:

  • 对于每一个任务, 现实中有大量边缘情况, 需要设计更复杂的规则来应对;
  • 对于每一个任务都需要人类专家设计规则, 但现实中复杂任务数不胜数, 且任务之间很难融洽;

Two Fundamental Challenges:

  • Ground natural language in robot state.

如何把自然语言转换成能够被机器人识别, 处理的状态?

例如抓取桌子最左边的红色方块, 需要机器人识别和定位到桌子, 桌子上的物体, 红色的方块, 以及相对位置关系.

  • Planning actions to solve a task.

如何规划合适的动作来完成任务?

例如”把货架的苹果放到桌子上”, 整个执行流程随着任务的复杂程度不断增加;

同样的, 多个物体的task planning.

Microsoft提出了Robotics with ChatGPT的设计原则:

  1. 设计一个task-relevant的API Library, 这个库将high-level的自然语言指令转换为low-level的机器人动作.
  2. 给chatgpt设计prompt, 用来描述任务目标并且需要陈输出high-level的库, 这些prompt也会包含任务的约束;
  3. 用户通过实验结果不断优化prompt来优化chatgpt的输出;
  4. 如果用户对目前的chatgpt的输出感到满意, 则最后部署到实际的机器人上.

科研机器人设计思路

目标是设计出一个机器人, 能够泛化的执行一些科研任务, 例如做材料实验, 化学实验等;

LLM在观测后输出一整个计划的行动序列, 就需要LLM对环境的未来走向有很好的了解.

Tricks for Improving Plan

  • 执行前Self-refine

在LLM生成plan后, 使用另一个LLM (或者人类设定的启发式评估函数) 来评估其好坏, 并且给出指导性意见;

但在这种情况下, LLM常常无法正确定位出错误; 通常会出现自己误导自己的情况;

  • Self-consistency

LLM通过多次生成规划并且选择最优的生成结果.

  • 执行后Self-reflection

让LLM从环境的反馈+自身反思中优化计划;

环境的反馈可能包括以下失败的原因:

  • 生成的动作不合规范, 没有正确调用action function;
  • 没有满足动作的执行条件, 如手中没有物体时, 执行放下物体的动作;
  • LLM自己以为生成的plan已经完成了任务, 实际任务还没有完成;
  • 生成的plan有很多冗余动作, 导致动作序列太长;

Replan机制: 结合ReAct与Planning

生成plan之后, 先执行plan获得环境的反馈和新观测信息后, 从当前状态继续规划;
同时, Replan允许LLM分析解决问题, 如可以先执行探索plan, 再根据结果调整和生成后续plan.

Tree search

如果计划未通过self-evaluation或执行失败, 在计划中的各步思考有哪些可行解决方案;

使用DFS或BFS, 在叶节点进行self-evaluation, 最终选择一条最优计划;

Tree search的缺点是

  • 消耗的tokens数大大增加;
  • 容易在错误的分支上花费很多搜索;

State tracking

预测出操作一步之后, 环境会发生什么样的变化, 对下一步的状态进行估计;

原本LLM应该包含这一步, 但是由于大模型的幻觉, 可能对下一步的状态估计不准确;

可以使用另一个LLM作为world model来预测每一步动作后的观测;

问题分析

  • 不着急直接输出Plan, 先让LLM自己分析一下这个问题: 这个问题有什么特点, 有什么易错点, 可以借助什么知识;
  • Step back: 退一步看问题, 先针对问题中的核心概念提一个相关问题, 再回答原问题;
  • 让LLM重新强调一下输入环境和规则的重点, 有助于提取关键信息和进一步思考;

Plan with Code

当再LLM的API输入high-level action之后得到的low-level actions sequence, 还可以让LLM直接生成代码, 例如生成Python代码;

其优点有:

  • 利用代码的逻辑能力和可直接执行性;
  • 有更高的自由性, 不只是调用已有的high-level action;
  • 有更好的复用性, 可以利用他人的代码以及保存已有代码;

甚至可以让LLM在code中自行定义有用的function;

Replan机制

如果计划遇到问题, 可以重新来过然后优化计划, 但是现实中常常不能重新来, 要在犯错后继续规划来完成任务.

需要做的事情:

  • 环境返回执行错误的原因,以及当前最新状态和观测
  • 让LLM解释犯错的原因

可以使用LLM worker, 监视每一步的执行结果, 或者在Plan code里面设定一些checkpoint, 保证环境变量按照计划执行;

A Inspector based on Markov Chain for low-level action sequence (An Innovative Idea)

马尔可夫链数学基础参考教程:

  • Introduction to Markov Chains

由于LLM以及大模型存在的幻觉, low-level action sequence本身的action或者sequence是有可能出错的;

对于像生物, 材料, 化学等实验任务, 必须保证实验的正确性和严谨性, 同时由于这些理科实验, 必须满足严格的逻辑, 因此借助马尔可夫链来检验low-level action sequence的正确性;
